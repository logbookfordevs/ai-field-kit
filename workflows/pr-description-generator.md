# Guide to Generating Effective Pull Request Descriptions

This guide outlines a structured process for generating a comprehensive and clear Pull Request (PR) description by analyzing the changes between two branches. A well-written PR description is crucial for efficient code reviews, better team collaboration, and creating a clear historical record.

## The Process

The process involves having the AI analyze the code changes and then structure them into a standardized template.

### Step 1: Identify the Branches

First, specify the two branches involved:

- **Base Branch (Target):** `_________________________` (e.g., `development`, `main`)
- **Feature Branch (Source):** `_________________________` (e.g., `feat/new-feature`, `fix/bug-123`)

### Step 2: Identify the Scope of Change

The AI will get a list of all changed files to understand the scope of the PR.

```bash
git diff --name-only <base_branch>...<feature_branch>
```

### Step 3: Analyze the Changes

The AI will then read the content of the changed files to understand the "what" and "why" of the modifications. It will analyze the diffs to synthesize a summary of changes, focusing on:

- The core purpose of the changes.
- New features added.
- Bugs fixed.
- Refactoring performed.
- Dependencies changed.

### Step 4: Generate the PR Description

Based on the analysis, the AI will generate a complete PR description using the template below. The goal is to provide a clear, concise, and context-rich summary that helps reviewers understand the changes quickly.

---

## Pull Request Description Template

This is the template that will be filled out.

### Title: [Type]: Brief description of the change

**Examples:**
- `feat: Add user authentication via JWT`
- `fix: Correct calculation error in payment processing`
- `refactor: Simplify database query logic in reporting module`
- `docs: Update API documentation for the /users endpoint`

---

### Summary

*A clear and concise summary of the change. Explain the "why" behind the PR. What problem is being solved? What is the business context? This section should also include any important context for testing the feature, such as specific configurations or environments.*

### Type of Change

*Please check the relevant option.*

- [ ] **Bug Fix** (a non-breaking change that fixes an issue)
- [ ] **New Feature** (a non-breaking change that adds functionality)
- [ ] **Breaking Change** (a fix or feature that would cause existing functionality to not work as expected)
- [ ] **Refactoring** (a code change that neither fixes a bug nor adds a feature)
- [ ] **Documentation** (changes to documentation only)
- [ ] **Chore** (updating build tasks, package manager configs, etc.)

### Related Issues

*Link to any relevant tickets, issues, or cards (e.g., JIRA-123, closes #45).*

---

### Changes Made

*A detailed, point-by-point list of the technical changes made. This section is generated by analyzing the file diffs. If the changes in a file are not immediately obvious from the diff, the AI will ask for a brief summary of what was changed in that file.*

- **`path/to/file_A.ts`**:
  - Added a new function `calculateTotal()` to handle complex calculations.
  - Refactored `processData()` for better readability.
- **`path/to/file_B.ts`**:
  - Fixed a null pointer exception in the `getUser()` method.
- **`config/env.example`**:
  - Added new environment variable `NEW_API_ENDPOINT`.

### Testing Considerations

*Provide clear, step-by-step instructions on how a reviewer can manually test and verify your changes, focusing on the user's journey and key scenarios. Include any necessary setup or configuration.*

-   **Scenario 1: [Brief description of scenario]**
    -   Verify [specific outcome 1].
    -   Confirm [specific outcome 2].
-   **Scenario 2: [Brief description of scenario]**
    -   Check [specific outcome 1].
    -   Ensure [specific outcome 2].
-   **Permissions/Access:**
    -   Verify that [feature/component] is accessible/restricted based on [user role/privilege].
-   **Regression:**
    -   Confirm that existing functionalities related to these changes remain unaffected.

### Purpose

*Explain the overall goal and objectives of this PR. What problem does it solve, or what new capability does it introduce?*

### Potential Issues

*Highlight any known issues, potential side effects, or areas of concern that reviewers should be aware of.*

### Additional Notes

*Any other relevant information, context, or considerations for the reviewer. This could include design decisions, future work, or acknowledgments.*
